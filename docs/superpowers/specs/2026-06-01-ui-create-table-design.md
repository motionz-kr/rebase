# UI 기반 테이블 추가 (Create Table) — 설계

작성일: 2026-06-01

## 1. 목적

스키마(데이터베이스) 노드를 우클릭해 **새 테이블을 UI로 생성**한다. 자주 쓰는 컬럼 속성은 폼으로 빠르게 채우고, 생성될 `CREATE TABLE` SQL을 **직접 편집**할 수 있게 해서 폼에 없는 모든 기능(외래키·복합 제약·인덱스·엔진 옵션 등)도 SQL을 손봐 실행할 수 있게 한다. 즉 "CREATE TABLE로 할 수 있는 모든 것"을 UI에서 보장한다.

이는 기존 [UI 기반 테이블/컬럼 편집](2026-06-01-ui-ddl-table-editing-design.md) 기능의 확장이며, `ddlBuilder` / `runDdl` / 컨텍스트 메뉴 패턴을 재사용한다.

## 2. 범위

### 포함 (v1)
- DB(스키마) 노드 우클릭 → **테이블 추가…** 메뉴
- 컬럼 폼 속성: 이름 · 타입 · NULL 허용 · 기본값 · **PK** · **자동증가** · **UNIQUE**
- 복합 기본키(여러 컬럼 PK 체크) → 테이블 레벨 `PRIMARY KEY (...)`
- **편집 가능한 SQL 미리보기**: 폼이 SQL을 생성하되, 사용자가 직접 편집하면 수동 모드로 전환되어 폼이 덮어쓰지 않음. 실행은 화면의 SQL 그대로.
- 대상 방언: **MySQL, PostgreSQL** (Redis 해당 없음)

### 제외 (v1, YAGNI)
- 외래키 · 복합 유니크 · 체크 제약 · 인덱스 · 파티션 · 엔진/charset 등을 위한 **전용 폼 컨트롤** — 이런 고급 기능은 폼이 아니라 **SQL 직접 편집**으로 처리한다(완전성 메커니즘).
- 편집 가능한 SQL 미리보기는 이번 **테이블 추가**에만 적용. 기존 수정/삭제 다이얼로그(`TableEditDialog`/`TableActionDialog`)는 변경하지 않는다.

## 3. 핵심 결정 사항

1. **프론트엔드 전용.** 백엔드(Go) 변경 없음. `CREATE TABLE`은 정책 분류상 비파괴·쓰기이므로 기존 `runDdl`(allowWrite+confirmDestructive)로 실행된다(파괴 플래그는 무시됨).
2. **별도 컴포넌트** `CreateTableDialog`. 기존 `TableEditDialog`의 diff 로직과 성격이 달라(생성은 baseline·diff가 없음) 섞지 않는다.
3. **DB 노드 전용 컨텍스트 메뉴**(`dbMenu` 상태). 기존 테이블 컨텍스트 메뉴(`menu`)와 분리한다.
4. **편집 가능한 SQL이 완전성을 보장한다.** 폼은 편의 스캐폴드이고, 텍스트 영역의 SQL이 최종 실행 대상이다.
5. 시작 시 `id BIGINT, PK, 자동증가` 행을 미리 채우되 자유롭게 수정·삭제 가능한 시작점이다(강제 아님).

## 4. 구성요소

### 4.1 순수 빌더 — `apps/renderer/src/lib/ddlBuilder.ts`에 추가 (TDD)

```ts
export interface CreateColumnSpec {
  name: string;
  type: string;
  nullable: boolean;
  defaultValue?: string;
  primaryKey?: boolean;
  autoIncrement?: boolean;
  unique?: boolean;
}

export function buildCreateTable(
  driver: Driver,
  table: string,
  columns: CreateColumnSpec[],
): string[];
```

생성 규칙(한 문장, 여러 줄로 포맷):
```
CREATE TABLE <qt> (
  <colDef>,
  ...,
  [PRIMARY KEY (<pk cols>)]
)
```
- `colDef` = `<qname> <type>[ NOT NULL][ AUTO_INCREMENT | GENERATED BY DEFAULT AS IDENTITY][ UNIQUE][ DEFAULT <v>]`
- **PK 컬럼은 `NOT NULL` 강제**(MySQL 요구, PostgreSQL도 PK는 NOT NULL 함의).
- **자동증가**: MySQL `AUTO_INCREMENT`(NOT NULL 뒤), PostgreSQL `GENERATED BY DEFAULT AS IDENTITY`. 자동증가 컬럼엔 `DEFAULT`를 붙이지 않는다.
- **UNIQUE**: 컬럼 레벨 `UNIQUE`. (복합 유니크는 SQL 편집으로.)
- PK가 1개 이상이면 컬럼들을 모아 테이블 레벨 `PRIMARY KEY (<q col1>, ...)` 절을 마지막에 추가. PK가 없으면 절 생략.
- 식별자 인용은 기존 `quoteIdent` 재사용. 들여쓰기 2칸, 읽기 좋게.
- 유효한 컬럼(이름·타입 비어있지 않음)만 포함. 컬럼이 하나도 없으면 빈 배열 반환(상위에서 실행 비활성).

예시:
```sql
-- MySQL
CREATE TABLE `t` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `email` VARCHAR(255) NOT NULL UNIQUE,
  PRIMARY KEY (`id`)
)
-- PostgreSQL
CREATE TABLE "t" (
  "id" bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
  "email" text NOT NULL UNIQUE,
  PRIMARY KEY ("id")
)
```

### 4.2 UI — `apps/renderer/src/components/CreateTableDialog.tsx` (신규)

Props: `{ profileId, driver: 'mysql'|'postgres', database, onClose, onApplied }`.

상태:
- `rows: Row[]` — 편집 행(이름·타입·nullable·defaultValue·primaryKey·autoIncrement·unique). 시작값: `id BIGINT, pk, autoIncrement` 한 행.
- `tableName: string` — 새 테이블 이름.
- `sql: string` — 미리보기/실행 SQL.
- `dirty: boolean` — 사용자가 SQL을 수동 편집했는지.
- `running`, `error`.

동작:
- `!dirty`일 때 폼(`rows`+`tableName`)이 바뀌면 `buildCreateTable`로 `sql`을 재생성한다(단일 문장이므로 `[0]`).
- 사용자가 SQL 텍스트 영역을 편집하면 `dirty=true` → 이후 폼 변경은 `sql`을 덮어쓰지 않는다.
- **[폼 기준으로 다시 생성]** 버튼: `dirty=false`로 되돌리고 폼에서 `sql` 재생성(수동 편집 폐기).
- **실행**: `runDdl(profileId, sql.trim() ? [sql] : [])`. 성공 → `onApplied()`+`onClose()`. 실패 → 엔진 에러 표시, 다이얼로그 유지.
- 실행 버튼 비활성 조건: `sql.trim()===''` 또는 `running` 또는 `tableName.trim()===''`(폼 모드일 때).

레이아웃: 테이블 이름 입력 → 컬럼 표(PK·자동증가·UNIQUE 체크박스 포함, 행 추가/삭제) → 편집 가능한 SQL 텍스트 영역(+[폼 기준으로 다시 생성]) → 에러 → 취소/실행.

### 4.3 SchemaExplorer 배선 — `apps/renderer/src/components/SchemaExplorer.tsx` 수정

- DB 노드 `tree-row`([SchemaExplorer.tsx:210](apps/renderer/src/components/SchemaExplorer.tsx#L210))에 `onContextMenu` 추가 → `setDbMenu({ x, y, db })`.
- `dbMenu` 상태 신설(기존 `menu`와 분리), 바깥 클릭/Esc 닫기는 기존 effect에 통합.
- DB 컨텍스트 메뉴: **테이블 추가…** 한 항목 → `setCreate({ db })` 후 메뉴 닫기.
- `{create && <CreateTableDialog ... key={create.db} onApplied={() => refreshAfterDdl(create.db)} />}` 마운트. `driver`는 `driver as 'mysql'|'postgres'`로 좁힘.
- 성공 후 `refreshAfterDdl(db)`로 트리·자동완성 갱신(기존 함수 재사용).

## 5. 데이터 흐름

```
DB 노드 우클릭 → dbMenu → 테이블 추가… → CreateTableDialog(create.db)
  폼 편집(컬럼/PK/AI/UNIQUE/이름) → buildCreateTable → SQL 텍스트영역(!dirty일 때 자동)
  [사용자가 SQL 직접 편집] → dirty=true (폼이 더 이상 덮어쓰지 않음)
    → 실행 → runDdl(profileId, [sql])  [allowWrite]
      → 성공: onApplied() → refreshAfterDdl(db) + 닫기
      → 실패: 엔진 에러 표시, 유지
```

## 6. 에러 처리

- 빈 테이블 이름/컬럼 없음 → 실행 비활성.
- 잘못된 SQL/엔진 에러(예: 중복 테이블, 문법 오류) → 엔진 메시지를 다이얼로그에 표시하고 닫지 않음.
- 수동 편집 SQL은 사용자 책임(폼 검증 우회) — 그래서 엔진 에러를 그대로 노출하는 것이 중요.

## 7. 테스트

- **`ddlBuilder.test.ts`에 `buildCreateTable` 케이스 추가**(TDD, RED 먼저):
  - 두 방언 × PK+자동증가 기본 케이스
  - UNIQUE 컬럼
  - 복합 PK(여러 컬럼) → 단일 `PRIMARY KEY (a, b)`
  - PK 없음 → `PRIMARY KEY` 절 생략
  - 기본값 있는 컬럼
  - PK 컬럼은 NOT NULL 강제
  - 자동증가 컬럼엔 DEFAULT 미부여
  - 컬럼 없음 → 빈 배열
- **다이얼로그/편집-SQL** — `dev-mysql`, `verify-pg` CDP 실제 검증: 폼으로 테이블 생성 → 미리보기 확인 → 실행 → 트리에 새 테이블 등장. SQL 직접 편집(예: 컬럼 추가)해서 실행 → 반영. 생성한 테이블은 검증 후 DROP으로 정리(특히 `dev-mysql`은 사용자 DB).
- **백엔드/Go 변경 없음.**
